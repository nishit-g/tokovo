# The Tokovo App Bible 📖

**Building Production-Grade Apps for the Tokovo Ecosystem**

This is the definitive guide to creating apps for Tokovo. It covers the philosophy, architecture, and step-by-step implementation details required to build "Best-in-Class" applications.

---

## 🏛️ Core Philosophy

Tokovo apps are **Simulations**, not just UIs. They must adhere to three core principles:

1.  **Resolution Independence**: Apps must look perfect on *any* device (iPhone, Pixel, Tablet) without code changes.
2.  **Sandbox Architecture**: Apps are "Content Only". They must never render system UI (Status Bars, Bezels).
3.  **Strict State/Logic Separation**: The Runtime (Data) is completely separate from the UI (Presentation).

---

## 🏗️ Architecture: The "Sandbox" Model

We follow a strict **Host vs. Guest** architecture to ensure scalability and reusability.

| Layer | Responsibility | Example |
| :--- | :--- | :--- |
| **Renderer (Host)** | Handles the "Physical World". Renders the device frame, status bar, and manages safe areas. | `TokovoRenderer`, `DeviceFrame` |
| **App (Guest)** | Handles the "Logical Content". Renders the inner UI of the app. agnostic of the device it's running on. | `WhatsappChatView`, `TwitterUI` |

**❌ DON'T:**
*   Render a Status Bar inside your app.
*   Import `iPhone16Profile` directly into your app.
*   Hardcode pixel values like `390px` width.

**✅ DO:**
*   Accept `width`, `height`, and `safeAreaInsets` as props.
*   Use `AppSurface` to scale your logical design to the physical container.

## 🚀 Step-by-Step Implementation

### Phase 1: Setup

Create a new package in `packages/apps-<name>`. Use the standard structure:

```text
packages/apps-myapp/
├── package.json          # Dependencies (@tokovo/core)
├── src/
│   ├── index.ts          # Public Exports
│   ├── types.ts          # Semantic Types (The "Truth")
│   ├── runtime.ts        # Reducer & Event Handling
│   ├── ui.tsx            # The Visual Layer
│   └── components/       # Platform-Specific UI
│       ├── ios/
│       └── android/
```

### Phase 2: Define Truth (`types.ts`)

Start with **Data**, not UI. Define strict Discriminated Unions for your content.

```typescript
// types.ts
export type MessageType = "text" | "image" | "system";

export interface BaseMessage {
    id: string;
    from: string;
    timestamp: number;
}

export interface TextMessage extends BaseMessage {
    type: "text";
    text: string;
}

export interface ImageMessage extends BaseMessage {
    type: "image";
    url: string;
    aspectRatio: number;
}

export type AppMessage = TextMessage | ImageMessage;
```

### Phase 4: Resolution Independence (`ui.tsx`)

This is the most critical step. You must map your **Logical Design** to the **Physical Screen**.

1.  **Pick a Logic Width**: We standardize on **393px** (iPhone 14/15/16 Pro width). Design everything assuming this width.
2.  **Receive Props**: Your main component receives physical dimensions from the Renderer.
3.  **Scale**: Use `AppSurface` to glue them together.

```tsx
// ui.tsx
import { AppSurface } from "@tokovo/core";

export interface MyAppProps {
    width?: number;          // Physical Width (e.g. 1179)
    height?: number;         // Physical Height (e.g. 2556)
    safeAreaInsets?: Insets; // Physical Safe Areas
}

export const MyApp: React.FC<MyAppProps> = ({ width, height, safeAreaInsets }) => {
    // 1. Define Design Baseline
    const DESIGN_WIDTH = 393;
    
    // 2. Calculate Scale & Logical Safe Areas
    const targetWidth = width || 1179;
    const scale = targetWidth / DESIGN_WIDTH;
    
    const safeTop = (safeAreaInsets?.top || 0) / scale;
    const safeBottom = (safeAreaInsets?.bottom || 0) / scale;

    return (
        <AppSurface
            designWidth={DESIGN_WIDTH}
            targetWidth={targetWidth}
            targetHeight={height || 2556}
        >
            <div style={{ paddingTop: safeTop, paddingBottom: safeBottom }}>
                {/* 
                    Now write Standard CSS! 
                    fontSize: 16 means 16 "logical" pixels.
                    It will render as 48 physical pixels on iPhone 16.
                */}
                <Header />
                <Content />
            </div>
        </AppSurface>
    );
};
```

### Phase 5: Semantic Grouping

Don't just map over arrays. Process your data into **Semantic Groups** using hooks.

**The "WhatsApp Pattern":**
1.  **Raw Data**: Array of messages ordered by time.
2.  **Hook (`useMessageGrouping`)**:
    *   Groups consecutive messages from same sender.
    *   Adds `isFirst`, `isLast`, `hasTail` metadata.
    *   Injects time dividers if gap > 1 hr.
3.  **Render**: Map the *Groups*, not the messages.

```tsx
// hooks/useMessageGrouping.ts
export function useMessageGrouping(messages: Message[]) {
    return useMemo(() => {
        const groups = [];
        // ... heavy grouping logic ...
        return groups;
    }, [messages]);
}
```

This keeps your UI code clean ("dumb") and your business logic testable.

## 💅 Styling & Assets

### 1x Logical Pixels
Always use **Logical Pixels** in your CSS.
*   Font Size `17px` (Standard Body) -> Renders as `51px` on device.
*   Border Radius `20px` -> Renders as `60px`.

### Platform Segregation
If your app supports both iOS and Android, use a **Strategy Pattern**.

```tsx
const PlatformComponents = {
    ios: { Header: IOSHeader, List: IOSList },
    android: { Header: AndroidHeader, List: AndroidList }
};
```

Determine the platform via props and select the correct component set at runtime.

## 🧪 Verification

Never assume it looks good. Verify it.

1.  **Video Runner**: Use the `video-runner` app to see your app running inside a real device frame.
2.  **Timeline Ops**: Trigger DSL events (`message.receive`, `scroll`) to verify dynamic behavior.
3.  **Cross-Device**: Switch the profile in your test episode (`iphone16` -> `pixel`) to verify your scaling logic holds up.

## 🧹 Checklist for PRs

- [ ] **Strict Types**: Are `runtime.ts` and `types.ts` strictly typed?
- [ ] **Pure UI**: Does `ui.tsx` accept `width/height` props? (No direct device imports)
- [ ] **AppSurface**: Is `AppSurface` used with `designWidth={393}`?
- [ ] **No Bezel**: Did you check that you aren't rendering a Status Bar?
- [ ] **Grouping**: Is complex list logic moved to a custom hook?
