# Data Flow

Tokovo uses a deterministic, multi-stage pipeline to transform high-level episode descriptions into frame-perfect video output. Each stage produces immutable intermediate representations that enable caching, debugging, and parallel processing.

## Pipeline Overview

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           TOKOVO DATA PIPELINE                               │
└─────────────────────────────────────────────────────────────────────────────┘

   AUTHORING              COMPILATION                 RUNTIME
   ─────────              ───────────                 ───────

┌──────────┐         ┌──────────────┐         ┌──────────────────┐
│   DSL    │         │   Compiler   │         │      Engine      │
│ (fluent) │────────▶│  (lowering)  │────────▶│   (handlers)     │
└──────────┘         └──────────────┘         └──────────────────┘
     │                      │                         │
     ▼                      ▼                         ▼
┌──────────┐         ┌──────────────┐         ┌──────────────────┐
│   IR     │         │ RuntimeEvent │         │   WorldState     │
│TrackEvent│         │   Timeline   │         │  (per-frame)     │
└──────────┘         └──────────────┘         └──────────────────┘
                                                      │
                                                      ▼
                                              ┌──────────────────┐
                                              │    Renderer      │
                                              │ (React/Remotion) │
                                              └──────────────────┘
                                                      │
                                                      ▼
                                              ┌──────────────────┐
                                              │   Video Frame    │
                                              │    (canvas)      │
                                              └──────────────────┘
```

## Stage 1: DSL → IR (TrackEvent)

The fluent DSL captures user intent in a readable, chainable API. Each method call produces a `TrackEvent` in the Intermediate Representation (IR).

```typescript
// DSL (authoring layer)
episode
  .at(0).openApp("whatsapp")
  .at(30).send("Hey!")
  .at(60).bgm("upbeat-track").play()
  .at(90).voice("segment-1").play();

// Produces IR TrackEvents:
[
  { kind: "DEVICE", type: "OPEN_APP", at: 0, payload: { appId: "whatsapp" } },
  { kind: "CHAT", type: "SEND_MESSAGE", at: 30, payload: { text: "Hey!" } },
  { kind: "AUDIO", type: "BGM_START", at: 60, payload: { soundId: "upbeat-track" } },
  { kind: "VOICE", type: "PLAY_SEGMENT", at: 90, payload: { segmentId: "segment-1", ... } },
]
```

### IR Event Kinds

| Kind           | Purpose              | Examples                                       |
| -------------- | -------------------- | ---------------------------------------------- |
| `DEVICE`       | Device state changes | `OPEN_APP`, `LOCK`, `SET_BADGE`                |
| `CHAT`         | Messaging events     | `SEND_MESSAGE`, `RECEIVE_MESSAGE`, `TYPING`    |
| `AUDIO`        | Sound and music      | `BGM_START`, `PLAY_SOUND`, `STOP`, `CROSSFADE` |
| `VOICE`        | Voice-over segments  | `PLAY_SEGMENT`, `STOP_VOICE`                   |
| `CAMERA`       | Camera movements     | `PAN`, `ZOOM`, `SHAKE`                         |
| `CALL`         | Phone calls          | `INCOMING_CALL`, `CALL_ANSWERED`, `CALL_ENDED` |
| `NOTIFICATION` | System notifications | `SHOW_NOTIFICATION`                            |

## Stage 2: Compiler (Lowering)

The compiler transforms high-level `TrackEvents` into low-level `RuntimeEvents`. This "lowering" process:

1. Resolves references (sound IDs → file paths)
2. Calculates frame timing
3. Expands macros (typing indicators, read receipts)
4. Applies deterministic defaults

```typescript
// TrackEvent (high-level)
{ kind: "AUDIO", type: "BGM_START", at: 60, payload: { soundId: "upbeat" } }

// Lowered to RuntimeEvent (low-level)
{ kind: "AUDIO", type: "PLAY", at: 60, soundId: "upbeat", bus: "music", loop: true, volume: 0.35 }
```

### Lowering Routes

| Input Kind | Lowering Function    | Output RuntimeEvent                      |
| ---------- | -------------------- | ---------------------------------------- |
| `DEVICE`   | `lowerDeviceEvent()` | `OPEN_APP`, `LOCK`, `SET_BADGE`          |
| `CHAT`     | `lowerChatEvent()`   | `SEND_MESSAGE` + derived typing/receipts |
| `AUDIO`    | `lowerAudioEvent()`  | `PLAY`, `STOP`, `FADE_OUT`, `CROSSFADE`  |
| `VOICE`    | `lowerVoiceEvent()`  | Voice `PLAY_SEGMENT`, `STOP_VOICE`       |
| `CAMERA`   | `lowerCameraEvent()` | `SET_CAMERA`, `ANIMATE_CAMERA`           |

## Stage 3: Engine (Handlers)

The engine processes `RuntimeEvents` in frame order, updating `WorldState` through pure handler functions. Each event kind has a dedicated handler.

```typescript
// Engine loop (simplified)
function computeWorldState(events: RuntimeEvent[], frame: number): WorldState {
  let world = createInitialWorldState();

  for (const event of events.filter((e) => e.at <= frame)) {
    const handler = getHandler(event.kind);
    world = handler(event, world);
  }

  return world;
}
```

### Handler Registry

| RuntimeEvent Kind | Handler                      | State Modified                         |
| ----------------- | ---------------------------- | -------------------------------------- |
| `DEVICE`          | `processDeviceEvent()`       | `devices[id].currentApp`, `isLocked`   |
| `AUDIO`           | `processAudioEvent()`        | `audio.activeSounds`, `audio.musicBed` |
| `VOICE`           | `processVoiceEvent()`        | `audio.activeSounds` (voice bus)       |
| `CAMERA`          | `processCameraEvent()`       | `camera.zoom`, `camera.pan`            |
| `NOTIFICATION`    | `processNotificationEvent()` | `devices[id].notifications`            |

## Stage 4: WorldState

`WorldState` is the complete, deterministic snapshot of the video at a given frame. It's computed fresh for each frame to ensure determinism.

```typescript
interface WorldState {
  frame: number;
  devices: Record<string, DeviceState>;
  camera: CameraState;
  audio: AudioState;
  anchors: AnchorState;
}

interface AudioState {
  activeSounds: Record<string, SoundCue>;
  buses: {
    music: AudioBusConfig;
    ui: AudioBusConfig;
    sfx: AudioBusConfig;
    voice: AudioBusConfig;
  };
  musicBed?: MusicBed;
  outgoingMusicBed?: MusicBed; // For crossfade transitions
}
```

## Stage 5: Renderer

The renderer consumes `WorldState` and produces React components that Remotion captures as video frames.

```typescript
// Renderer flow
import { createTokovoRegistries, getConfig } from "@tokovo/core";
import { createDeviceRegistries, registerDevicesPlugin } from "@tokovo/devices";

const tokovoRegistries = createTokovoRegistries();
const deviceRegistries = createDeviceRegistries();
registerDevicesPlugin(tokovoRegistries, deviceRegistries);

const rendererRegistries = {
  plugins: tokovoRegistries.plugins,
  devices: deviceRegistries,
};

function EpisodeRenderer({ frame }: { frame: number }) {
  const world = runEpisode(episode, frame, {
    mode: "preview",
    registries: tokovoRegistries.engine,
    config: getConfig(),
  });

  return (
    <RendererRegistryProvider registries={rendererRegistries}>
      <>
        <CameraLayer camera={world.camera}>
          {Object.entries(world.devices).map(([id, device]) => (
            <DeviceFrame key={id} device={device} />
          ))}
        </CameraLayer>
        <AudioLayer audio={world.audio} frame={frame} />
      </>
    </RendererRegistryProvider>
  );
}
```

## Audio Pipeline Detail

Audio has a specialized sub-pipeline within the engine:

```
┌─────────────────────────────────────────────────────────────────┐
│                     AUDIO SUB-PIPELINE                          │
└─────────────────────────────────────────────────────────────────┘

 DSL                  IR                 Compiler              Engine
 ───                  ──                 ────────              ──────

.bgm("track")    BGM_START      ─────▶   PLAY (music bus)    ─────▶  musicBed
.play("click")   PLAY_SOUND     ─────▶   PLAY (sfx bus)      ─────▶  activeSounds
.voice("seg")    PLAY_SEGMENT   ─────▶   VOICE PLAY_SEGMENT  ─────▶  activeSounds (voice bus)
.crossfade(...)  CROSSFADE      ─────▶   CROSSFADE           ─────▶  musicBed + outgoingMusicBed

                                              │
                                              ▼
                                      ┌───────────────┐
                                      │   Policies    │
                                      ├───────────────┤
                                      │ • Spam gate   │
                                      │ • Concurrency │
                                      │ • Priority    │
                                      │ • Ducking     │
                                      └───────────────┘
                                              │
                                              ▼
                                      ┌───────────────┐
                                      │    Mixer      │
                                      ├───────────────┤
                                      │ • Bus gains   │
                                      │ • Duck calc   │
                                      │ • Fade apply  │
                                      └───────────────┘
                                              │
                                              ▼
                                      ┌───────────────┐
                                      │  AudioLayer   │
                                      │  (Remotion)   │
                                      └───────────────┘
```

### Audio Buses

| Bus     | Purpose                  | Max Concurrent | Default Gain |
| ------- | ------------------------ | -------------- | ------------ |
| `music` | Background music, BGM    | 1              | 0.35         |
| `ui`    | UI sounds (taps, swipes) | 3              | 0.9          |
| `sfx`   | Sound effects            | 4              | 0.8          |
| `voice` | Voice-over narration     | 1              | 1.0          |

### Ducking

When a sound with a `duck` rule plays, other buses are automatically attenuated:

```typescript
// Voice ducks music by 70%
{
  soundId: "narration-01",
  bus: "voice",
  duck: { targetBus: "music", amount: 0.7, attack: 5, release: 15 }
}
```

## Determinism Guarantees

Every stage is designed for determinism:

| Stage    | Guarantee                                 |
| -------- | ----------------------------------------- |
| DSL      | Produces identical IR for identical calls |
| Compiler | Pure functions, no side effects           |
| Engine   | Immutable state updates via Immer         |
| Renderer | React strict mode compatible              |

This enables:

- **Scrubbing**: Jump to any frame instantly
- **Caching**: Memoize expensive computations
- **Debugging**: Reproduce any frame exactly
- **Parallelism**: Render frames in any order
