# Creating Custom Anchors

> **Gain precise control over camera movements with semantic anchors that stay deterministic**

Anchors are named reference points that the camera can focus on during episode playback. Tokovo anchors are **defined by plugins** and **computed from world state**, so they stay deterministic and replayable in every render.

**What you'll learn:**

- How Tokovo's anchor system actually works today
- Defining anchors in your plugin with `PluginAnchorRegistry`
- Returning normalized bounds (0–1) for stability across devices
- Building dynamic anchors from world state
- Framing tips for cinematic camera moves
- Common failure modes and fixes

**Prerequisites:**

- Completed [First Episode Quickstart](/getting-started/first-episode)
- Familiarity with [Cinematic Camera](/guides/cinematic-camera)
- Basic understanding of plugin structure

**Time estimate:** 20-30 minutes

## What Are Anchors?

Anchors are **named camera targets**. Instead of panning to raw pixels, you reference a semantic name like `app_myapp:message` and the camera resolves it to a bounding box.

**Benefits:**

- ✅ **Semantic** — `focus("app_myapp:message")` communicates intent
- ✅ **Stable** — normalized bounds adapt to different device profiles
- ✅ **Deterministic** — no DOM access, no runtime layout drift
- ✅ **Reusable** — anchor names work across episodes

## How The Anchor System Works

### Anchor Registration Flow

```
┌────────────────────┐
│  Plugin Definition │  anchors.providers
└────────┬───────────┘
         │
         ▼
┌────────────────────┐
│ PluginManagerClass │  registers providers
└────────┬───────────┘
         │
         ▼
┌────────────────────┐
│ Anchor Registry    │  appId + anchorName
└────────┬───────────┘
         │
         ▼
┌────────────────────┐
│ Camera System      │  cam.focus("app_x:anchor")
└────────────────────┘
```

### Important Constraints

- **Providers are pure**: `provider(world, deviceId) => AnchorBounds | null`
- **Bounds are normalized**: all values are **0–1**, not pixels
- **Anchor IDs are namespaced**: `appId:anchorName`

## Creating Simple Anchors

### 1. Define a Plugin Anchor Registry

Create a file for anchor providers (pattern used in core plugins):

```typescript
// packages/apps-myapp/src/runtime/adapters/anchors.ts
import type { PluginAnchorRegistry, WorldState } from "@tokovo/core";

export const MyAppAnchors: PluginAnchorRegistry = {
  providers: {
    header: (_world: WorldState, _deviceId: string) => ({
      x: 0.5,
      y: 0.12,
      width: 0.9,
      height: 0.18,
    }),

    input: (_world: WorldState, _deviceId: string) => ({
      x: 0.5,
      y: 0.88,
      width: 0.95,
      height: 0.18,
    }),
  },
};
```

**Notes:**

- `x` and `y` are **center coordinates** (0–1)
- `width` and `height` are **normalized** (0–1)

### 2. Attach Anchors to the Plugin

```typescript
// packages/apps-myapp/src/plugin.ts
import type { TokovoPluginContract } from "@tokovo/core";
import { MyAppAnchors } from "./runtime/adapters/anchors";

export const MyAppPlugin: TokovoPluginContract<"app_myapp"> = {
  id: "app_myapp",
  version: "1.0.0",
  displayName: "My App",
  reducer: myAppReducer,
  views: { AppRoot: MyAppView },
  createInitialState: createMyAppInitialState,
  anchors: MyAppAnchors,
};
```

### 3. Use Anchors in Camera DSL

```typescript
episode("story", { duration: 30, fps: 30 })
  .camera((cam) => {
    cam.at("0s").focus("app_myapp:header");
    cam.at("2s").focus("app_myapp:input");
  });
```

## Dynamic Anchors (World State Driven)

Anchors should update based on **state**, not DOM. The provider uses `WorldState` so you can compute bounds from your app state.

### Example: Anchor the Latest Message

```typescript
import type { PluginAnchorRegistry, WorldState } from "@tokovo/core";

type MyAppState = {
  messages: Array<{
    id: string;
    anchor: { x: number; y: number; width: number; height: number };
  }>;
};

export const MyAppAnchors: PluginAnchorRegistry = {
  providers: {
    message: (world: WorldState) => {
      const app = world.appState.app_myapp as MyAppState | undefined;
      const last = app?.messages?.[app.messages.length - 1];
      return last?.anchor ?? null;
    },
  },
};
```

**Pattern:**

- Store layout-derived anchors in your app state
- Update them through events or reducers
- Keep providers pure and deterministic

## Framing Overrides (Cinematic Control)

Anchors define **where** the camera looks. Framing defines **how** it composes the shot.

```typescript
import type { PluginAnchorRegistry } from "@tokovo/core";

export const MyAppAnchors: PluginAnchorRegistry = {
  providers: {
    message: (_world, _deviceId) => ({ x: 0.5, y: 0.55, width: 0.85, height: 0.2 }),
  },
  framing: {
    message: {
      anchorPoint: { x: 0.5, y: 0.55 },
      paddingPx: 40,
      targetFill: 0.6,
    },
  },
};
```

**Use framing when:**

- You want a tighter crop (`targetFill`)
- The camera should bias up/down/left/right (`anchorPoint`)
- You want extra breathing room (`paddingPx`)

## Naming Conventions

**Recommended:**

- Anchor names are **local** to the plugin (`"message"`, `"header"`)
- Camera references use the **namespaced key** (`"app_myapp:message"`)
- Keep names short and semantic

**Avoid:**

- Raw positions in the name (`"x120_y400"`)
- Anchors that depend on DOM measurement

---

## Best Practices

1. **Use normalized bounds** — always return values in 0–1 space
2. **Keep providers pure** — no async, no DOM, no randomness
3. **Namespaced focus** — always include `appId:` in camera focus
4. **Prefer state-driven geometry** — store geometry in app state
5. **Define framing where it matters** — not every anchor needs custom framing

## Troubleshooting

### Anchor Is Not Found

**Symptoms:** camera jumps to default framing or nothing happens.

**Checklist:**

- Plugin is registered in `apps/video-runner/src/runtime.ts`
- Anchor provider returns non-null bounds
- Camera uses namespaced key (`app_myapp:anchor`)

### Anchor Appears in Wrong Position

**Symptoms:** camera focuses off-screen or too large.

**Checklist:**

- Bounds are normalized (0–1), not pixels
- `x` and `y` are **center**, not top-left
- `width` and `height` are normalized

### Anchor Jitters Frame-to-Frame

**Symptoms:** camera moves subtly each frame.

**Checklist:**

- Provider is deterministic
- You are not using unstable values
- State updates are consistent and event-driven
