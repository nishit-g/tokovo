# Building an App Plugin

An app plugin should encapsulate app-specific behavior so `@tokovo/core` and `@tokovo/renderer` do not need app-aware branching.

## A Good App Plugin Owns

- runtime reducer and initial state
- snapshot schema and initial-view schema
- snapshot/view version migrations
- bootstrap validation and hydration
- track-builder or authoring helpers
- React views
- layout and anchor integration
- optional audio rules

## A Good App Plugin Does Not Own

- control-plane data storage
- generic device behavior
- global runtime boot logic

## Bootstrap Contract

If a plugin accepts frame-0 app data, it must define:

- `bootstrap.snapshot.currentVersion`
- `bootstrap.snapshot.validate(...)`
- `bootstrap.snapshot.migrate(...)` when versions advance
- `bootstrap.view.currentVersion`
- `bootstrap.view.validate(...)`
- `bootstrap.view.migrate(...)` when versions advance
- `bootstrap.hydrate(...)`

The compiler validates and migrates snapshot/view entries before calling `hydrate()`. Compiler-owned app-history seeding is not supported.

The runtime app, usually `video-runner`, should register plugins explicitly.
