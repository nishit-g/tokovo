## Why Use the DSL?

### DSL vs Raw IR

The Tokovo engine consumes **Intermediate Representation (IR)** — a low-level event structure. You could write IR by hand:

```json
{
  "events": [
    {
      "t": 30,
      "kind": "APP",
      "deviceId": "phone",
      "appId": "app_whatsapp",
      "type": "MESSAGE_RECEIVED",
      "payload": { "from": "Alice", "content": "Hey!" }
    }
  ]
}
```

**Problems with raw IR:**

- ❌ No type safety (easy to make typos)
- ❌ Manual frame calculations (30fps × 1s = frame 30)
- ❌ Verbose and repetitive
- ❌ Hard to refactor timing
- ❌ No code reuse

**Benefits of the DSL:**

- ✅ **Type-safe** — TypeScript catches errors at compile time
- ✅ **Expressive** — `.at('1s').receive('Alice', 'Hey!')` is readable
- ✅ **Reusable** — Extract patterns into functions
- ✅ **Timing abstractions** — Use `'1s'`, `'500ms'`, or raw frames
- ✅ **Compiler-validated** — Prevents invalid event sequences

The DSL is your **high-level language** for storytelling. The compiler translates it into IR for the engine.

## Basic Structure Review

Every Tokovo episode follows this structure:

```typescript
import { episode } from "@tokovo/dsl";

const myEpisode = episode("story-name", { duration: 30, fps: 30 })
  .device("phone", "iphone16", (device) => {
    // Configure device (apps, state)
    device.app("app_whatsapp", (wa) => {
      // Schedule app events
      wa.at("1s").receive("Alice", "Hello!");
      wa.at("3s").send("You", "Hi!");
    });
  })
  .camera((cam) => {
    // Choreograph camera movements
    cam.at("0s").focus("deviceScreen");
    cam.at("1s").focus("lastMessage");
  })
  .build();
```

**Key Components:**

1. **Episode metadata** — `episode('name', { duration, fps })`
2. **Device configuration** — `.device(id, profile, builder)`
3. **App track** — `.app(appId, builder)` schedules app events
4. **Camera track** — `.camera(builder)` choreographs focus/zoom/shake
5. **Build step** — `.build()` compiles to IR

Now let's explore advanced patterns for complex episodes.

## Organizing Complex Episodes

### Separate Concerns by File

As episodes grow, extract conversations and logic into separate modules.

**File Structure:**

```
episodes/
  breakup-story/
    index.ts              # Main episode definition
    conversations.ts      # Message sequences
    camera.ts             # Camera choreography
    constants.ts          # Timing constants
```

**`conversations.ts`** — Reusable message patterns:

```typescript
import type { WhatsAppTrackBuilder } from '@tokovo/dsl'

export const openingExchange = (wa: WhatsAppTrackBuilder) => {
  wa.at('1s').receive('Sarah', 'We need to talk...')
  wa.at('5s').typing('Sarah')
  wa.at('8s').receive('Sarah', 'I think we should break up.')
}

export const desperateResponse = (wa: WhatsAppTrackBuilder) => {
  wa.at('12s').typing('You')
  wa.at('15s').send('You', 'What? Why?')
  wa.at('18s').typing('You')
  wa.at('20s').send('You', 'Please, let's talk about this.')
}
```

**`camera.ts`** — Separate camera logic:

```typescript
import type { CameraTrackBuilder } from "@tokovo/dsl";

export const breakupCameraWork = (cam: CameraTrackBuilder) => {
  cam.at("0s").zoom(0.9);
  cam.at("0s").focus("deviceScreen");
  cam.at("1s").focus("lastMessage");
  cam.at("8s").shake(1.2);
  cam.at("8.5s").zoom(1.4);
  // ... more choreography
};
```

**`index.ts`** — Clean composition:

```typescript
import { episode } from "@tokovo/dsl";
import { openingExchange, desperateResponse } from "./conversations";
import { breakupCameraWork } from "./camera";

export const breakupEpisode = episode("breakup", { duration: 30, fps: 30 })
  .device("phone", "iphone16", (d) => {
    d.app("app_whatsapp", (wa) => {
      openingExchange(wa);
      desperateResponse(wa);
    });
  })
  .camera(breakupCameraWork)
  .build();
```

**Why this works:**

- **Maintainability** — Each file has a single responsibility
- **Testability** — Test conversations and camera separately
- **Reusability** — Import patterns into other episodes
- **Readability** — Main file shows high-level structure

## Reusable Pattern Libraries

Extract common interaction patterns into functions.

### Pattern: Receive → Typing → Send

A fundamental conversation pattern:

```typescript
const receiveTypingSend = (
  app: WhatsAppTrackBuilder,
  from: string,
  incomingMsg: string,
  yourReply: string,
  startTime: number,
  thinkTime = 2, // seconds to "think" before replying
  typeTime = 2, // seconds to "type" reply
) => {
  app.at(`${startTime}s`).receive(from, incomingMsg);
  app.at(`${startTime + thinkTime}s`).typing("You");
  app.at(`${startTime + thinkTime + typeTime}s`).send("You", yourReply);
};
```

**Usage:**

```typescript
d.app("app_whatsapp", (wa) => {
  receiveTypingSend(wa, "Alice", "Want coffee?", "Sure!", 1);
  receiveTypingSend(wa, "Alice", "When?", "Now?", 7);
  receiveTypingSend(wa, "Alice", "See you in 10!", "Perfect!", 13);
});
```

**Benefits:**

- Consistent timing across conversations
- Easy to adjust thinkTime/typeTime globally
- Reduces copy-paste errors

### Pattern: Multi-Message Burst

Send multiple messages in quick succession:

```typescript
const burst = (
  app: WhatsAppTrackBuilder,
  from: string,
  messages: string[],
  startTime: number,
  interval = 1, // seconds between messages
) => {
  messages.forEach((msg, i) => {
    app.at(`${startTime + i * interval}s`).receive(from, msg);
  });
};
```

**Usage:**

```typescript
burst(wa, 'Alice', [
  'OMG',
  'You won't believe what happened',
  'I got the job!',
  '🎉🎉🎉'
], 5, 0.8) // Start at 5s, 0.8s between messages
```

### Pattern: Back-and-Forth Conversation

Alternate between two speakers:

```typescript
const backAndForth = (
  app: WhatsAppTrackBuilder,
  person1: string,
  person2: string,
  exchanges: [string, string][], // [person1 msg, person2 reply]
  startTime = 0,
  pauseBetween = 3, // seconds between exchanges
) => {
  let t = startTime;

  exchanges.forEach(([msg1, msg2]) => {
    app.at(`${t}s`).receive(person1, msg1);
    t += pauseBetween / 2;
    app.at(`${t}s`).typing(person2);
    t += 1;
    app.at(`${t}s`).send(person2, msg2);
    t += pauseBetween / 2;
  });
};
```

**Usage:**

```typescript
backAndForth(
  wa,
  "Alice",
  "You",
  [
    ["Hey! How are you?", "Good! You?"],
    ["Great! Want to grab lunch?", "Sure, when?"],
    ["1pm?", "Perfect!"],
  ],
  0,
  4,
);
```

### Pattern: Delayed Reaction with Typing

Show hesitation before replying:

```typescript
const hesitateAndReply = (
  app: WhatsAppTrackBuilder,
  message: string,
  delay: number, // seconds before starting to type
  startTime: number,
) => {
  app.at(`${startTime}s`).typing("You");
  app.at(`${startTime + 0.5}s`).stopTyping("You"); // Pause
  app.at(`${startTime + delay}s`).typing("You");
  app.at(`${startTime + delay + 2}s`).send("You", message);
};
```

**Usage:**

```typescript
wa.at("10s").receive("Sarah", "Do you still love me?");
hesitateAndReply(wa, "Of course I do.", 3, 12); // 3s hesitation
```

## Timeline Management

### Absolute vs Relative Timing

**❌ Bad: Hardcoded absolute times**

```typescript
wa.at("1s").receive("Alice", "Hi");
wa.at("3s").receive("Alice", "How are you?");
wa.at("5s").receive("Alice", "Want to hang out?");
wa.at("7s").receive("Alice", "Let me know!");
```

**Problem:** Adjusting timing requires changing every timestamp.

**✅ Good: Use variables for flexibility**

```typescript
const START = 1;
const PAUSE = 2;

wa.at(`${START}s`).receive("Alice", "Hi");
wa.at(`${START + PAUSE}s`).receive("Alice", "How are you?");
wa.at(`${START + PAUSE * 2}s`).receive("Alice", "Want to hang out?");
wa.at(`${START + PAUSE * 3}s`).receive("Alice", "Let me know!");
```

**Better:** Now changing `PAUSE` adjusts all timing proportionally.

**✅ Even Better: Accumulator pattern**

```typescript
let t = 1; // Start time

wa.at(`${t}s`).receive("Alice", "Hi");
t += 2;

wa.at(`${t}s`).receive("Alice", "How are you?");
t += 2;

wa.at(`${t}s`).receive("Alice", "Want to hang out?");
t += 2;

wa.at(`${t}s`).receive("Alice", "Let me know!");
```

**Why this works:** Easy to insert new messages without recalculating all timestamps.

### Duration Calculation

Calculate total episode duration from component lengths:

```typescript
// constants.ts
export const TIMINGS = {
  intro: 3,
  conversation: 15,
  cameraSetup: 2,
  outro: 3,
};

const totalDuration = Object.values(TIMINGS).reduce((sum, t) => sum + t, 0);

// index.ts
export const myEpisode = episode("story", { duration: totalDuration, fps: 30 });
```

**Benefits:**

- Self-documenting timing structure
- Easy to adjust individual sections
- Prevents duration mismatches

### Timeline Constants Pattern

Group related timings:

```typescript
const TIMELINE = {
  openingMessage: 1,
  firstReply: 4,
  plot_twist: 10,
  emotional_peak: 15,
  resolution: 20,
};

wa.at(`${TIMELINE.openingMessage}s`).receive("Alice", "Hey");
wa.at(`${TIMELINE.firstReply}s`).send("You", "Hi!");
wa.at(`${TIMELINE.plot_twist}s`).receive(
  "Alice",
  "I need to tell you something...",
);
cam.at(`${TIMELINE.plot_twist}s`).shake(1.2);
```

**Why this works:**

- Named timestamps are self-documenting
- Easy to synchronize camera and app events
- Refactoring timing is trivial

## Type Safety Best Practices

### Always Type Your Builders

```typescript
import type { WhatsAppTrackBuilder, CameraTrackBuilder } from "@tokovo/dsl";

// ✅ Good — TypeScript will catch errors
const myConversation = (wa: WhatsAppTrackBuilder) => {
  wa.at("1s").receive("Alice", "Hello");
  // wa.at('2s').invalidMethod() // ❌ TypeScript error!
};

const myCameraWork = (cam: CameraTrackBuilder) => {
  cam.at("0s").focus("deviceScreen");
  // cam.at('1s').invalidAnchor() // ❌ TypeScript error!
};
```

**Benefits:**

- Autocomplete for available methods
- Compile-time error detection
- Refactoring safety

### Use Discriminated Unions for Variations

```typescript
type ConversationVariant =
  | { type: 'friendly'; name: string }
  | { type: 'formal'; title: string; lastName: string }

const greeting = (wa: WhatsAppTrackBuilder, variant: ConversationVariant) => {
  if (variant.type === 'friendly') {
    wa.at('1s').receive(variant.name, 'Hey! What's up?')
  } else {
    wa.at('1s').receive(
      `${variant.title} ${variant.lastName}`,
      'Good afternoon. I hope this message finds you well.'
    )
  }
}

// Usage
greeting(wa, { type: 'friendly', name: 'Alice' })
greeting(wa, { type: 'formal', title: 'Dr.', lastName: 'Smith' })
```

### Validate Timing with Types

```typescript
type Seconds = number; // Semantic type alias

const scheduleMessage = (
  app: WhatsAppTrackBuilder,
  time: Seconds,
  from: string,
  message: string,
) => {
  if (time < 0) {
    throw new Error("Time cannot be negative");
  }
  app.at(`${time}s`).receive(from, message);
};
```

## Testing Episodes

### Unit Test Conversation Patterns

```typescript
import { describe, it, expect } from "vitest";
import { compile } from "@tokovo/compiler";
import { episode } from "@tokovo/dsl";
import { receiveTypingSend } from "./patterns";

describe("receiveTypingSend pattern", () => {
  it("generates correct event sequence", () => {
    const ep = episode("test", { duration: 10, fps: 30 })
      .device("phone", "iphone16", (d) => {
        d.app("app_whatsapp", (wa) => {
          receiveTypingSend(wa, "Alice", "Hi", "Hello", 1, 2, 2);
        });
      })
      .build();

    const ir = compile(ep);

    // Should have 3 events: receive, typing, send
    expect(ir.events).toHaveLength(3);
    expect(ir.events[0].type).toBe("MESSAGE_RECEIVED");
    expect(ir.events[1].type).toBe("TYPING_START");
    expect(ir.events[2].type).toBe("MESSAGE_SENT");
  });
});
```

### Integration Test: State at Specific Time

```typescript
import { replay } from "@tokovo/core";

describe("Breakup Episode", () => {
  it("shows correct state at emotional peak", () => {
    const ir = compile(breakupEpisode);
    const state = replay(ir, 8); // 8 seconds in

    // Should have Sarah's breakup message
    expect(state.apps.app_whatsapp.messages).toHaveLength(2);
    expect(state.apps.app_whatsapp.messages[1].content).toBe(
      "I think we should break up.",
    );

    // Camera should be focused and zoomed
    expect(state.camera.focus).toBe("lastMessage");
    expect(state.camera.zoom).toBeGreaterThan(1.0);
  });
});
```

### Snapshot Testing for Regressions

```typescript
it("matches snapshot for full episode IR", () => {
  const ir = compile(myEpisode);
  expect(ir).toMatchSnapshot();
});
```

**Benefits:**

- Detect unintended changes in compiled IR
- Ensure refactoring doesn't break episodes
- Document expected event structure

### Test Timing Calculations

```typescript
describe("Timeline calculations", () => {
  it("fits within episode duration", () => {
    const TIMELINE = {
      intro: 3,
      conversation: 15,
      outro: 5,
    };

    const totalDuration = Object.values(TIMELINE).reduce(
      (sum, t) => sum + t,
      0,
    );

    expect(totalDuration).toBeLessThanOrEqual(30); // Episode duration
  });
});
```

## Best Practices Summary

### ✅ Do This

1. **Extract reusable patterns** — DRY (Don't Repeat Yourself)
2. **Use timing variables** — Avoid hardcoded timestamps
3. **Type your builders** — Leverage TypeScript's type system
4. **Separate concerns** — Conversations, camera, constants in separate files
5. **Test critical paths** — Unit test patterns, integration test state
6. **Comment complex timing** — Explain "why" for non-obvious delays
7. **Keep episodes under 60s** — Performance and attention span

### ❌ Avoid This

1. **Magic numbers** — Use named constants for timing
2. **Copy-pasting conversations** — Extract into functions
3. **Hardcoding device IDs** — Pass as parameters
4. **Skipping type annotations** — Lose autocomplete and safety
5. **Overly complex builders** — Break into smaller functions
6. **Untested episodes** — Bugs are hard to spot in compiled IR

## Common Anti-Patterns to Avoid

### ❌ The Copy-Paste Conversation

```typescript
// ❌ Bad — repetitive, hard to maintain
wa.at("1s").receive("Alice", "Hey!");
wa.at("3s").typing("You");
wa.at("5s").send("You", "Hi!");

wa.at("7s").receive("Bob", "Hey!");
wa.at("9s").typing("You");
wa.at("11s").send("You", "Hi!");

wa.at("13s").receive("Carol", "Hey!");
wa.at("15s").typing("You");
wa.at("17s").send("You", "Hi!");
```

**✅ Fix: Extract pattern**

```typescript
const greetingExchange = (wa, from, startTime) => {
  wa.at(`${startTime}s`).receive(from, "Hey!");
  wa.at(`${startTime + 2}s`).typing("You");
  wa.at(`${startTime + 4}s`).send("You", "Hi!");
};

greetingExchange(wa, "Alice", 1);
greetingExchange(wa, "Bob", 7);
greetingExchange(wa, "Carol", 13);
```

### ❌ The Timing Labyrinth

```typescript
// ❌ Bad — impossible to adjust timing
wa.at("1.234s").receive("Alice", "Message 1");
wa.at("3.789s").receive("Alice", "Message 2");
wa.at("6.123s").receive("Alice", "Message 3");
wa.at("9.456s").receive("Alice", "Message 4");
```

**✅ Fix: Use accumulator or constants**

```typescript
let t = 1;
wa.at(`${t}s`).receive("Alice", "Message 1");
t += 2.5;
wa.at(`${t}s`).receive("Alice", "Message 2");
t += 2.5;
wa.at(`${t}s`).receive("Alice", "Message 3");
t += 2.5;
wa.at(`${t}s`).receive("Alice", "Message 4");
```

### ❌ The God Builder

```typescript
// ❌ Bad — 500-line builder function
d.app("app_whatsapp", (wa) => {
  // ... 500 lines of messages, typing, reactions ...
});
```

**✅ Fix: Separate into acts/scenes**

```typescript
const act1 = (wa) => {
  /* ... */
};
const act2 = (wa) => {
  /* ... */
};
const act3 = (wa) => {
  /* ... */
};

d.app("app_whatsapp", (wa) => {
  act1(wa);
  act2(wa);
  act3(wa);
});
```

### ❌ The Untyped Mystery

```typescript
// ❌ Bad — no types, error-prone
const doSomething = (app, data) => {
  app.at(`${data.time}s`).receive(data.from, data.msg);
};
```

**✅ Fix: Add explicit types**

```typescript
interface MessageData {
  time: number;
  from: string;
  msg: string;
}

const scheduleMessage = (app: WhatsAppTrackBuilder, data: MessageData) => {
  app.at(`${data.time}s`).receive(data.from, data.msg);
};
```

## Advanced: Builder Composition

Combine multiple patterns into higher-level builders:

```typescript
const dramaticReveal = (
  wa: WhatsAppTrackBuilder,
  cam: CameraTrackBuilder,
  from: string,
  buildup: string[],
  reveal: string,
  startTime: number
) => {
  let t = startTime

  // Buildup messages
  buildup.forEach((msg) => {
    wa.at(`${t}s`).receive(from, msg)
    cam.at(`${t}s`).focus('lastMessage')
    t += 2
  })

  // Typing suspense
  wa.at(`${t}s`).typing(from)
  cam.at(`${t}s`).focus('typingIndicator')
  cam.at(`${t}s`).zoom(1.2)
  t += 3

  // The reveal
  wa.at(`${t}s`).receive(from, reveal)
  cam.at(`${t}s`).focus('lastMessage')
  cam.at(`${t}s`).shake(1.5)
  cam.at(`${t + 0.2}s`).zoom(1.5)
}

// Usage
episode('mystery', { duration: 20, fps: 30 })
  .device('phone', 'iphone16', (d) => {
    d.app('app_whatsapp', (wa) => {})
  })
  .camera((cam) => {
    dramaticReveal(
      wa,
      cam,
      'Detective',
      ['I've been investigating...', 'And I found something.'],
      'You're the killer.',
      5
    )
  })
```

**Note:** This requires passing multiple builders. Alternative: return event data to be applied later.

## Performance Considerations

### Keep Episodes Under 60 Seconds

- Remotion rendering time scales with duration
- Viewer attention span
- Easier to debug shorter episodes

### Avoid Excessive Events

```typescript
// ❌ Bad — 1000 events in 30 seconds
for (let i = 0; i < 1000; i++) {
  wa.at(`${i * 0.03}s`).receive("Spam", "Message");
}

// ✅ Good — Reasonable event density
wa.at("1s").receive("Alice", "Hello");
wa.at("5s").receive("Alice", "How are you?");
// ... 10-20 events total
```

**Guideline:** 1-2 events per second maximum for smooth playback.
