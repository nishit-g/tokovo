## Performance Philosophy

Tokovo's performance model is built on three principles:

### **1. Determinism = Predictable Performance**

Because `runEpisode()` is a pure, incremental function, performance is predictable:

```ts
// Same inputs = same computational cost
const state1 = runEpisode(episode, t, opts); // 5ms
const state2 = runEpisode(episode, t, opts); // 5ms (always)
```

No cache invalidation bugs. No unpredictable GC pauses from leaked listeners.

### **2. Frame Independence**

Each frame is computed independently. This means:

- No accumulated state leaks across frames
- Parallel rendering possible (Remotion's strength)
- Frame scrubbing is instant (no "warm up" needed)

### **3. Optimization is Data-Driven**

Performance bottlenecks live in your episode data:

- Too many events → slow `runEpisode()`
- Deep state nesting → expensive Immer operations
- Large text blocks → slow layout calculations

Optimize the **episode structure**, not the engine.

## Determinism Benefits for Performance

Tokovo's deterministic design directly improves performance:

### **Snapshot Caching**

Because `runEpisode()` is deterministic, we can cache state snapshots:

```ts
// Incremental replay with a StateCache
const stateCache = createStateCache(30);
const keyframedIndex = createKeyframedEventIndex(episode.events, 30);

// When rendering frame 148:
const state = runEpisode(episode, 148, {
  ...opts,
  keyframedEventIndex: keyframedIndex,
  stateCache,
}); // Only replays delta frames
```

**Performance impact:** 3-5x faster rendering for 30s+ episodes.

**How it works:**

1. During first render pass, save state snapshots every N frames
2. On subsequent renders, start from nearest snapshot instead of frame 0
3. Only replay events between snapshot and target frame

**Best for:**

- Long episodes (30s+)
- Dense event timelines (100+ events)
- Multi-pass rendering workflows

### **Memoization Works Reliably**

React memoization depends on deterministic inputs:

```ts
// This works because world reference changes predictably
const layout = useMemo(() => {
  return computeLayout(world, t);
}, [world, t]);
```

**Anti-pattern (breaks memoization):**

```ts
// ❌ BAD: Math.random() changes on every render
const delay = Math.random() * 1000;
const layout = useMemo(() => computeLayout(world, delay), [world, delay]);
// Memoization useless - delay always different
```

**Best practice:**

```ts
// ✅ GOOD: Deterministic input from episode data
const delay = event.delay || 500;
const layout = useMemo(() => computeLayout(world, delay), [world, delay]);
// Memoization works - delay stable across renders
```

### **Parallel Rendering (Remotion)**

Remotion can render frames in parallel because Tokovo is stateless:

```bash
# Render 4 frames simultaneously
remotion render --concurrency=4
```

Each worker calls `runEpisode(episode, t, opts)` independently. No shared state = no race conditions.

**Performance impact:** 4x faster renders on 4-core machines.

## Efficient Episode Design

### **1. Minimize Event Count**

Every event increases `runEpisode()` cost. Reduce unnecessary events:

#### ❌ **Anti-pattern: Event Spam**

```ts
// BAD: 100 events for smooth typing animation
for (let i = 0; i < text.length; i++) {
  track.at(`${i * 0.05}s`).typingProgress(i / text.length);
}
```

**Cost:** 100 events × 30 frames = 3,000 event lookups per second.

#### ✅ **Best Practice: Implicit Animation**

```ts
// GOOD: 2 events, derive progress from time
track.at("0s").typingStart();
track.at("5s").typingEnd(); // Layout engine interpolates progress
```

**Cost:** 2 events. Layout engine computes progress: `(t - startT) / duration`.

**Performance impact:** 50x fewer events for same visual result.

### **2. Batch State Changes**

Group related state changes into single events:

#### ❌ **Anti-pattern: Fragmented Updates**

```ts
// BAD: 3 separate events
track.at("1s").updateStatus("typing");
track.at("1s").updateActiveConversation("conv_123");
track.at("1s").markAsRead("msg_456");
```

**Cost:** 3 Immer `produce()` calls = 3× overhead.

#### ✅ **Best Practice: Batched Event**

```ts
// GOOD: Single event with multiple mutations
track.at("1s").batchUpdate({
  status: "typing",
  activeConversation: "conv_123",
  markAsRead: "msg_456",
});
```

**Cost:** 1 Immer `produce()` call.

**Performance impact:** 3x fewer state mutations.

### **3. Limit Active Effects**

Camera effects have per-frame computation cost:

```ts
// Each effect processes every frame between startFrame and endFrame
camera.at("0s").shake({ duration: "10s", intensity: 5 }); // 300 frames @ 30fps
```

**Guidelines:**

- **Limit concurrent effects:** Max 3-4 active effects at once
- **Shorter durations:** Use 1-2s effects, not 10s+ (unless necessary)
- **Strategic timing:** Don't overlap 5 shake effects

#### Example: Optimized Effects

```ts
// ❌ BAD: 4 overlapping effects (300+ frame computations)
camera.at("0s").shake({ duration: "10s" });
camera.at("0s").zoom({ scale: 1.2, duration: "10s" });
camera.at("0s").dutchTilt({ angle: 5, duration: "10s" });
camera.at("0s").chromatic({ intensity: 3, duration: "10s" });

// ✅ GOOD: Sequential effects (30-60 frame computations each)
camera.at("0s").shake({ duration: "1s" });
camera.at("1.5s").zoom({ scale: 1.2, duration: "2s" });
camera.at("4s").dutchTilt({ angle: 5, duration: "1s" });
```

**Performance impact:** 5x fewer effect computations per frame.

## Event Count Reduction Strategies

### **Strategy 1: Use Spans Instead of Points**

#### ❌ Before: Many Point Events

```ts
track.at("1s").typingStart();
track.at("1.5s").typing();
track.at("2s").typing();
track.at("2.5s").typing();
track.at("3s").typingEnd();
```

**Cost:** 5 events.

#### ✅ After: Single Span Event

```ts
track.span("1s", "3s").typing(); // Layout engine interpolates
```

**Cost:** 1 event.

**Reduction:** 5 → 1 (80% fewer events).

### **Strategy 2: Derive from Time, Not Events**

If visual state can be computed from time, don't use events:

#### ❌ Before: Event-Driven Progress

```ts
// 10 events for battery drain
for (let i = 100; i >= 0; i -= 10) {
  track.at(`${(100 - i) * 0.5}s`).batteryLevel(i);
}
```

**Cost:** 11 events.

#### ✅ After: Computed Progress

```ts
// No events - compute in layout engine
function getBatteryLevel(t: number, startT: number, duration: number): number {
  const progress = (t - startT) / duration;
  return Math.max(0, 100 - progress * 100);
}
```

**Cost:** 0 events. Computed at render time.

**Reduction:** 11 → 0 (100% fewer events).

### **Strategy 3: Remove Redundant State Updates**

Only emit events when state **actually changes**:

#### ❌ Before: Redundant Updates

```ts
track.at("0s").appOpen("whatsapp");
track.at("1s").appOpen("whatsapp"); // Already open
track.at("2s").appOpen("whatsapp"); // Still open
```

**Cost:** 3 events (2 redundant).

#### ✅ After: State Change Only

```ts
track.at("0s").appOpen("whatsapp"); // State change
// No redundant events
```

**Cost:** 1 event.

**Reduction:** 3 → 1 (67% fewer events).

## Layout Engine Optimization

The layout engine runs **every frame**. Optimize hot paths:

### **1. Avoid Expensive Calculations in Loops**

#### ❌ **Anti-pattern: Repeated Work**

```ts
// BAD: Recompute height for every message
messages.forEach((msg) => {
  const height = computeHeight(msg.text, config); // Expensive
  y += height;
});
```

**Cost:** `O(n * m)` where `n` = messages, `m` = text length.

#### ✅ **Best Practice: Memoize Heights**

```ts
// GOOD: Cache heights by message ID
const heightCache = new Map<string, number>();

messages.forEach((msg) => {
  let height = heightCache.get(msg.id);
  if (!height) {
    height = computeHeight(msg.text, config);
    heightCache.set(msg.id, height);
  }
  y += height;
});
```

**Cost:** `O(n)` amortized (cache hits after first frame).

**Performance impact:** 10-20x faster for 50+ messages.

### **2. Use useMemo for Layout Computations**

Layout engine uses `useMemo` to avoid recomputation:

```ts
// packages/renderer/src/engines/useLayoutEngine.ts
export function useLayoutEngine(input: LayoutEngineInput): LayoutEngineOutput {
  return useMemo(() => {
    // Expensive layout computation
    const layout = computeLayout(world, t);
    return layout;
  }, [world, t]); // Only recompute when world/t change
}
```

**Why it works:** `world` reference changes **only** when state changes (Immer guarantees structural sharing).

**Performance impact:** Skips layout computation when world unchanged (e.g., during camera-only effects).

### **3. Limit Text Length in Height Calculations**

Approximate heights for long text:

```ts
// Fast approximation for tall messages
function estimateHeight(text: string, config: ChatLayoutConfig): number {
  const lines = Math.ceil(text.length / config.charsPerLine);

  // Cap at reasonable max (avoid computing 1000-line heights)
  const cappedLines = Math.min(lines, 50);

  return config.baseBubbleHeight + (cappedLines - 1) * config.lineHeight;
}
```

**Performance impact:** Constant-time height computation (vs. DOM measurement).

### **4. Avoid Deep State Nesting**

Immer's `produce()` cost scales with nesting depth:

#### ❌ **Anti-pattern: Deep Nesting**

```ts
// BAD: 5 levels deep
draft.devices.phone1.apps.whatsapp.conversations.conv1.messages.push(msg);
```

**Cost:** Immer must clone 5 nested objects.

#### ✅ **Best Practice: Flat Structure**

```ts
// GOOD: 2 levels deep
draft.appState.whatsapp.messages.push(msg);
```

**Cost:** Immer clones 2 objects.

**Performance impact:** 2-3x faster state updates.

## Remotion Rendering Tips

### Fast Local Renders (Tokovo Profile)

Tokovo ships a tuned "fast" render profile for local iteration on a MacBook:

```bash
EPISODE_ID=v2-enterprise-long-showcase pnpm --filter video-runner render:fast
```

What it does:
- Enables GPU rendering (`--gl=angle`) and hardware-accelerated encoding (`--hardware-acceleration=if-possible`).
- Uses max CPU concurrency for throughput.

For final exports, render without the fast profile or override flags to prioritize quality and predictable bitrate.

### **1. Use Concurrency for Faster Renders**

```bash
# Single-threaded (slow)
remotion render --output=video.mp4

# Multi-threaded (fast)
remotion render --concurrency=4 --output=video.mp4
```

**Performance impact:** 3-4x faster on multi-core machines.

**Best practice:** Set concurrency = CPU core count.

### **2. Optimize Image Assets**

#### Image Loading

Preload images to avoid per-frame fetching:

```ts
// BAD: Load image every frame (30 loads per second)
const img = new Image();
img.src = avatar.url; // Re-fetches every render

// GOOD: Preload once, use staticFile
import { staticFile } from "remotion";
const avatarSrc = staticFile("avatars/user1.png");
```

**Performance impact:** Eliminates network requests during rendering.

#### Image Sizing

Use appropriately sized images:

```ts
// ❌ BAD: 4K image for 100px avatar
<img src="avatar-4096x4096.png" width={100} height={100} />

// ✅ GOOD: Match image size to display size
<img src="avatar-200x200.png" width={100} height={100} />
```

**Performance impact:** Reduces decode time by 10-20x.

### **3. Limit Simultaneous Effects**

Remotion re-renders on every frame. Limit expensive effects:

#### High-Cost Effects

- Multiple blurs
- Large drop shadows
- Complex gradients
- Nested transforms

#### ❌ **Anti-pattern: Effect Overload**

```tsx
<div
  style={{
    filter: "blur(10px) drop-shadow(0 0 20px rgba(0,0,0,0.5))",
    background: "linear-gradient(45deg, ...20 color stops...)",
    transform: "translateX() rotateY() scale() skew()",
  }}
>
  {/* Nested children with more effects */}
</div>
```

**Cost:** GPU chokes on complex compositing.

#### ✅ **Best Practice: Selective Effects**

```tsx
// Use effects sparingly, on specific elements
<div style={{ transform: `scale(${scale})` }}>
  {/* Simple transforms only */}
</div>
```

**Performance impact:** 2-3x faster frame rendering.

### **4. Profile with Remotion Preview**

Use Remotion's preview tooling and React profiling:

```bash
pnpm --filter video-runner dev
```

Then:

1. Open React DevTools Profiler
2. Record during playback
3. Identify slow components
4. Optimize hot paths

**Look for:**

- Components rendering every frame (should memoize)
- Long render times (\u003e16ms @ 60fps)
- Unnecessary re-renders (useMemo/React.memo)

## Performance Checklist

Use this checklist before final renders:

### **Episode Structure**

- [ ] **Event count \u003c 200** for 30s episodes (reduce via spans/derivation)
- [ ] **No redundant events** (state changes only)
- [ ] **Batched state updates** (combine related changes)
- [ ] **Minimal deep nesting** in WorldState (flatten where possible)

### **Layout Engine**

- [ ] **Memoized height calculations** (cache by message ID)
- [ ] **useMemo for expensive computations**
- [ ] **Text length capped** in height estimates (\u003c50 lines)
- [ ] **No DOM measurement** (all layout pre-computed)

### **Camera Effects**

- [ ] **Max 3-4 concurrent effects**
- [ ] **Effect durations \u003c 3s** (unless cinematic intent)
- [ ] **No overlapping heavy effects** (shake + zoom + chromatic)

### **Assets**

- [ ] **Images preloaded** via staticFile()
- [ ] **Image sizes optimized** (match display size)
- [ ] **Fonts subset** (only needed glyphs)
- [ ] **No large video embeds** (use external refs)

### **Remotion Rendering**

- [ ] **Concurrency = CPU cores** (check with `--concurrency=4`)
- [ ] **1080p target** (4K only if needed)
- [ ] **30fps default** (60fps only for smooth motion)
- [ ] **Profiled in preview** (no components \u003e16ms)

### **Snapshot Caching**

- [ ] **Enabled for 30s+ episodes**
- [ ] **Interval = 30 frames** (balance memory vs. speed)
- [ ] **maxSnapshots = 100** (adjust based on memory)

## Common Performance Issues

### **Issue 1: Slow Rendering (Minutes for 30s Episodes)**

**Symptoms:**

- Remotion render takes \u003e 5 minutes for 30s episode
- High CPU usage during render
- Fans spin loudly

**Diagnosis:**

```bash
# Profile render
remotion render --log=verbose
```

**Common Causes:**

1. **Too many events:** > 500 events → slow `runEpisode()`
2. **Large images:** 4K+ images decoded every frame
3. **Complex effects:** Multiple blurs/shadows/gradients
4. **No concurrency:** Single-threaded rendering

**Fixes:**

- Reduce event count (use spans/derivation)
- Optimize image sizes
- Remove expensive effects
- Enable concurrency (`--concurrency=4`)

### **Issue 2: Janky Playback in Preview**

**Symptoms:**

- Preview stutters during playback
- Frame rate drops below 30fps
- Scrubbing is slow

**Diagnosis:**

Open React DevTools → Profiler → Record during playback.

**Common Causes:**

1. **Layout computation \u003e 16ms:** Expensive height calculations
2. **No memoization:** Components re-render every frame
3. **Deep component trees:** \u003e 100 components per frame

**Fixes:**

- Memoize layout computations (`useMemo`)
- Use `React.memo` for static components
- Flatten component hierarchy
- Cache height calculations

### **Issue 3: Memory Usage Grows During Render**

**Symptoms:**

- Render starts fast, slows over time
- Memory usage \u003e 4GB
- OOM errors on long episodes

**Diagnosis:**

```bash
# Monitor memory
node --max-old-space-size=8192 node_modules/.bin/remotion render
```

**Common Causes:**

1. **Snapshot cache unbounded:** No `maxSnapshots` limit
2. **Event accumulation:** Events not garbage collected
3. **Image cache leaks:** Images never released

**Fixes:**

- Set `maxSnapshots: 100` in cache config
- Clear snapshot cache between episodes
- Use weak references for image cache

## Advanced: Custom Snapshot Intervals

For very long episodes (60s+), tune snapshot intervals:

```ts
// High-frequency snapshots (better for scrubbing, more memory)
const cache = new SnapshotCache({
  interval: 10, // Every 10 frames (0.33s @ 30fps)
  maxSnapshots: 300, // 10s of snapshots
});

// Low-frequency snapshots (less memory, slower scrubbing)
const cache = new SnapshotCache({
  interval: 60, // Every 60 frames (2s @ 30fps)
  maxSnapshots: 50, // 100s of snapshots
});
```

**Trade-offs:**

- **Low interval (10):** Fast scrubbing, high memory usage
- **High interval (60):** Slow scrubbing, low memory usage
- **Sweet spot (30):** Balanced performance
