# Debugging with Tokovo Studio

> **Learn to debug episodes using Tokovo Studio's visual debugging tools**

Tokovo Studio is your debugging companion for episode development. It provides visual tools to inspect WorldState, scrub through timelines, preview camera movements, and diagnose issues before rendering. This guide covers current Studio capabilities and best practices for debugging deterministic episodes.

**What you'll learn:**

- How to launch and navigate Studio
- Timeline visualization and frame-by-frame debugging
- WorldState inspection techniques
- Camera preview and framing verification
- Event debugging strategies
- Common debugging workflows

**Prerequisites:**

- Completed [First Episode guide](/getting-started/first-episode)
- Understanding of [WorldState](/concepts/worldstate)
- Familiarity with [Timeline Events](/concepts/timeline-events)
- Basic knowledge of [Determinism](/concepts/determinism)

## What is Tokovo Studio?

Tokovo Studio is a **local development tool** for debugging episodes before committing to full Remotion renders. It combines:

- **Timeline visualization** — See all events in a visual timeline
- **Live preview** — Scrub through episodes frame-by-frame
- **State inspection** — Examine WorldState at any frame
- **Camera debugging** — Preview camera movements and anchors

Studio runs locally and integrates with your episode registry, loading episodes dynamically for debugging.

## Launching Studio

### Installation

Studio is included in the Tokovo monorepo:

```bash
# From project root
cd packages/studio

# Install dependencies (if needed)
pnpm install

# Start development server
pnpm dev
```

Studio will launch at `http://localhost:3000` (or the next available port).

### Accessing Episodes

Studio automatically loads episodes from `@tokovo/episodes` registry:

```typescript
// Episodes are auto-discovered via defineEpisode()
import { defineEpisode } from "@tokovo/episodes";

export const myEpisode = defineEpisode({
  meta: {
    id: "my-debug-episode",
    title: "Debugging Example",
    category: "test",
  },
  config: {
    format: "portrait",
    durationInFrames: 300, // 10s @ 30fps
  },
  build: (b) => {
    // Episode content
  },
});
```

Studio sees all registered episodes automatically.

## Timeline Visualization

### Event Tracks

Timeline events are organized by category in horizontal tracks:

**Track Types:**

| Track      | Event Kinds                                          | Color Code |
| ---------- | ---------------------------------------------------- | ---------- |
| **Camera** | Focus, Zoom, Shake, Animate, Track                   | Purple     |
| **Audio**  | BGM, SFX, Crossfade, Volume                          | Blue       |
| **App**    | Messages, Typing, Navigation, Reactions              | Green      |
| **Device** | Lock, Unlock, Open App, Close App                    | Orange     |
| **OS**     | Time, Battery, Notifications, System State           | Gray       |
| **Call**   | Incoming Call, Answer, Reject, Hang Up (Coming Soon) | Red        |

### Event Markers

Each event appears as a **marker** on the timeline at its `at` timestamp:

```
Camera Track:  ━━━●━━━━━━━●━━━━━━●━━━━
               0s  2s       5s      8s

● = Discrete event (at specific time)
━━━ = Span event (duration)
```

**Marker Styles:**

- **Dot (●)** — Discrete event (e.g., `.at('2s').focus('lastMessage')`)
- **Bar (━━━)** — Span event (e.g., `.span('2s', '5s').typing()`)
- **Hover** — Shows event details (type, payload, timestamp)

### Click-to-Seek

Click anywhere on the timeline to **jump to that frame**:

```typescript
// Click on timeline at 3.5s
// Preview updates immediately to frame 105 (3.5s × 30fps)
// WorldState recomputed via runEpisode(episode, 105)
```

This enables **frame-by-frame debugging** without waiting for renders.

## WorldState Inspection

### Current State View

> **Coming Soon**: Full WorldState inspector panel

Studio computes WorldState at the current frame using:

```typescript
import { runEpisode } from "@tokovo/core";

const world = runEpisode(episode, currentFrame, { mode: "preview" });
```

**What's Inspectable:**

- **Devices** — Lock state, foreground app, battery, time
- **App State** — Messages, conversations, typing indicators, navigation
- **Camera** — Current focus anchor, zoom level, shake state
- **Audio** — Active BGM, sound effects, volume levels
- **Notifications** — Active notifications, heads-up display state

### Frame-by-Frame Comparison

**Debugging Workflow:**

1. Scrub to frame where issue occurs
2. Inspect WorldState at that frame
3. Step backward/forward by single frames
4. Compare state changes between frames

```typescript
// Example debugging state transitions
Frame 89:  typing = true, messages = [msg1, msg2]
Frame 90:  typing = true, messages = [msg1, msg2, msg3] ← Message appears
Frame 91:  typing = false, messages = [msg1, msg2, msg3] ← Typing stops
```

This reveals **exactly when state changes** and helps identify event timing issues.

## Camera Preview

### Visual Camera Debugging

Studio shows the camera's perspective in real-time:

**Camera State:**

- **Focus Target** — Which anchor is focused (e.g., `lastMessage`, `deviceScreen`)
- **Zoom Level** — Current magnification (1.0 = normal, 1.5 = 150%)
- **Shake Intensity** — Active shake trauma level
- **Transitions** — Spring/easing animations in progress

**Why This Matters:**

Camera issues are hard to debug without visual feedback:

```typescript
// ❌ Without Studio: Guess if camera is framing correctly
cam.at("3s").focus("lastMessage");
cam.at("3s").zoom(1.3);

// ✅ With Studio: See exactly what the camera frames at 3s
// Scrub to frame 90, verify framing visually
```

### Anchor Verification

> **Coming Soon**: Anchor bounding box overlays

Verify that semantic anchors point to the correct UI elements:

```typescript
// Expected: Camera focuses on newest message
cam.at("5s").focus("lastMessage");

// Studio shows: Bounding box around "lastMessage" anchor
// If box is wrong, check app's anchor registration
```

This catches **anchor registration bugs** before rendering.

## Event Debugging

### Timeline Event Inspection

**Hover over event markers** to see full event details:

```typescript
// Hover on Camera marker at 2s:
{
  at: 60,           // Frame 60 (2s × 30fps)
  kind: "CAMERA",
  type: "FOCUS",
  payload: {
    anchor: "lastMessage",
    easing: "spring",
  }
}
```

**Use Cases:**

- Verify event timing (are events at expected frames?)
- Check event payloads (correct message text, camera targets)
- Find missing events (gaps in timeline visualization)

### Common Debugging Scenarios

#### Scenario 1: Message Doesn't Appear

**Symptoms:**

- Timeline shows message event at 3s
- Preview doesn't show message

**Debugging Steps:**

1. Scrub to frame where message should appear (frame 90)
2. Inspect WorldState → Check `world.appState.whatsapp.messages`
3. If message missing: Event didn't fire (check event `at` timestamp)
4. If message present but invisible: Layout issue (check `useLayoutEngine` output)

#### Scenario 2: Camera Not Focusing

**Symptoms:**

- Camera event fires but focus doesn't change

**Debugging Steps:**

1. Verify camera event is in timeline
2. Check anchor name (typo? `lastMessage` vs `lastmessage`)
3. Verify anchor exists (app must register anchors)
4. Check camera transition timing (may still be animating from previous focus)

#### Scenario 3: Events Out of Order

**Symptoms:**

- Message appears before typing indicator ends

**Debugging Steps:**

1. Check timeline visualization — are events in correct order?
2. Verify timestamps:
   ```typescript
   wa.at("2s").typing("Alice"); // Starts at frame 60
   wa.at("4s").receive("Alice", "Hello"); // Ends at frame 120
   // Gap: 2 seconds (60 frames) is correct
   ```
3. If order wrong: Adjust event `at` values

## Determinism Verification

Studio helps verify that episodes are **deterministic** (same frame always produces same output).

### Frame Consistency Test

**Workflow:**

1. Scrub to frame 150
2. Note exact WorldState (message count, camera position, etc.)
3. Jump to frame 0
4. Scrub back to frame 150
5. Verify WorldState is **identical**

**Why This Matters:**

Non-deterministic bugs cause:

- Different output on each render
- Remotion caching failures
- Unreliable video output

Common causes:

- Using `Math.random()` without seed
- Date.now() / timestamps
- Async operations without deterministic scheduling

Studio reveals these instantly via frame comparison.

## Debugging Workflows

### Workflow 1: Message Timing

**Goal:** Ensure messages appear at correct times with smooth transitions.

**Steps:**

1. Open episode in Studio
2. Play through conversation
3. Pause when message appears
4. Verify:
   - Message appears at expected frame
   - Typing indicator preceded it (if applicable)
   - Camera focuses on message (if choreographed)
5. Adjust timings in DSL if needed:
   ```typescript
   // Before debugging
   wa.at("2s").typing("Alice");
   wa.at("3s").receive("Alice", "Hello"); // Too fast
   // After debugging
   wa.at("2s").typing("Alice");
   wa.at("4s").receive("Alice", "Hello"); // Better pacing
   ```

### Workflow 2: Camera Choreography

**Goal:** Verify camera movements match narrative intent.

**Steps:**

1. Scrub through episode watching camera behavior
2. Pause at key moments (message reveals, shock reactions)
3. Check:
   - Focus targets correct UI elements
   - Zoom levels feel appropriate (not too tight/wide)
   - Shake events match emotional beats
4. Refine camera DSL:
   ```typescript
   // Before: Too subtle
   cam.at("5s").zoom(1.1);
   // After: More dramatic
   cam.at("5s").zoom(1.4);
   ```

### Workflow 3: Event Sequence Validation

**Goal:** Ensure complex event sequences execute correctly.

**Steps:**

1. Identify critical sequence (e.g., notification → unlock → open app → read message)
2. Scrub through sequence frame-by-frame
3. Verify each event fires in order
4. Check state transitions:
   ```
   Frame 0:   device.isLocked = true
   Frame 30:  notification appears
   Frame 60:  device.isLocked = false (unlocked)
   Frame 90:  device.foregroundApp = "app_whatsapp" (app opened)
   Frame 120: message marked as read
   ```

### Workflow 4: Audio Synchronization

**Goal:** Ensure audio events align with visual events.

**Steps:**

1. Play episode with audio
2. Verify BGM starts/ends at expected times
3. Check sound effects trigger on events:
   ```typescript
   // Message receives should sync with notification sound
   wa.at("3s").receive("Alice", "Hello");
   audio.at("3s").sfx("notification");
   ```
4. Verify crossfades are smooth (no abrupt cuts)

## Best Practices

### 1. Debug Early and Often

Don't wait for full renders to catch bugs:

```typescript
// ✅ Good workflow
1. Write episode DSL
2. Load in Studio
3. Scrub through timeline
4. Fix issues immediately
5. Render when verified

// ❌ Bad workflow
1. Write episode DSL
2. Render immediately
3. Watch 30s video to find 1-frame bug
4. Fix, re-render, repeat
```

Studio saves hours of render time.

### 2. Use Frame Numbers, Not Time

Frame numbers are more precise than seconds:

```typescript
// ❌ Hard to debug
cam.at("2.5s").focus("lastMessage"); // Frame 75 @ 30fps

// ✅ Easier in Studio
cam.at("2.5s").focus("lastMessage"); // Studio shows "Frame 75"
// Scrub to frame 75 to debug
```

### 3. Test Edge Cases

Use Studio to test unusual scenarios:

- **First frame (frame 0):** Initial state correct?
- **Last frame:** No events fire beyond duration?
- **Event boundaries:** Messages appear exactly at expected frames?
- **Simultaneous events:** Multiple events at same `at` timestamp work?

### 4. Compare Against Reference

Keep working episodes as references:

```typescript
// Open two browser tabs
Tab 1: Studio with buggy episode
Tab 2: Studio with working reference episode
// Compare timelines side-by-side
```

### 5. Document Findings

> **Coming Soon**: Export timeline screenshots and state snapshots

For now, manually document debugging findings:

```markdown
## Debug Log: Episode "breakup-scene"

- Frame 90: Camera should focus "lastMessage" but focuses "deviceScreen" instead
  - **Cause:** Typo in anchor name (`lastMessge` vs `lastMessage`)
  - **Fix:** Corrected anchor name in camera DSL
- Frame 150: Message appears 30 frames late
  - **Cause:** Typing span ends at frame 120, message at frame 150 (30 frame gap too long)
  - **Fix:** Changed message from `.at('5s')` to `.at('4s')`
```

## Troubleshooting

### Studio Won't Load Episode

**Symptoms:**

- Episode selector empty or episode missing

**Fixes:**

- Verify episode uses `defineEpisode()` (not raw IR)
- Check episode is exported from `@tokovo/episodes`
- Restart Studio dev server (`pnpm dev`)

### Preview Shows Black Screen

**Symptoms:**

- Timeline loads but preview area is black

**Fixes:**

- Check browser console for errors
- Verify all plugins registered:
  ```typescript
  const registries = createTokovoRegistries();
  const deviceRegistries = createDeviceRegistries();
  const pluginManager = new PluginManagerClass(registries.plugins);

  registerDevicesPlugin(registries, deviceRegistries);
  registerWhatsAppPlugin(pluginManager);
  registerNotificationPlugin(registries.engine);
  registerCameraPlugin(pluginManager, registries.engine);
  registerKeyboardPlugin(registries.engine);
  ```
- Verify episode compiles without errors

### Timeline Events Don't Match Preview

**Symptoms:**

- Events appear in timeline but preview doesn't reflect them

**Fixes:**

- Check event timestamps (negative values? beyond duration?)
- Verify event types are valid (typos in event names)
- Clear browser cache (stale compiled episode)

### Camera Preview Laggy

**Symptoms:**

- Preview stutters when scrubbing

**Fixes:**

- Close other browser tabs (reduce memory usage)
- Disable debug mode if enabled
- Reduce episode complexity (fewer concurrent events)

## Integration with Remotion

Studio previews use the **same rendering pipeline** as Remotion:

```typescript
// Studio preview (EpisodeRenderer.tsx)
const world = runEpisode(episode, frame, { mode: "preview" });
return <TokovoRenderer world={world} t={frame} />;

// Remotion render (video-runner)
const world = runEpisode(episode, frame, { mode: "render" });
return <TokovoRenderer world={world} t={frame} />;
```

**Why This Matters:**

- What you see in Studio = what renders in video
- No surprises when moving to production
- Deterministic preview = deterministic render

**Mode Differences:**

- `preview` mode: Optimized for scrubbing (caching, fewer calculations)
- `render` mode: Maximum quality (all effects, full precision)
