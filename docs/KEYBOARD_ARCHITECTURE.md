## 1. The Input Philosophy

In standard UI automation, typing is `input.value = "Hello"`.
In Tokovo, typing is a performance.

We simulate the **Physical Act of Typing**:
1.  **Ghost Touches:** Invisible fingers tapping specific coordinates on a virtual glass plane.
2.  **Layout Switching:** The engine must "press" the `123` key before it can type a number.
3.  **Visual Feedback:** Keys pop up (`IOSKeyboard`) or ripple (`AndroidKeyboard`) exactly when the event fires.
4.  **Variance:** No human types with 0ms variance. We use seeded RNG to add "human jitter" to every keystroke.

## 2. The Architecture Stack

How does `d.type("Hi 123")` working?

```mermaid
graph TD
    A[DSL: type('Hi 123')] -->|Expands to| B[Event Stream]
    B -->|Contains| C[KEY_DOWN 'H' -> KEY_UP 'H']
    B -->|Then| D[KEY_DOWN '123' (Switch Layout)]
    B -->|Then| E[KEY_DOWN '1' -> KEY_UP '1']
    E -->|Processed by| F[Keyboard Reducer]
    F -->|Updates| G[Keyboard State (Visible, CurrentKey, Layout)]
    G -->|Rendered by| H[KeyboardSurface (Scales to Device)]
    
    style A fill:#dae8fc
    style B fill:#e1d5e7
    style G fill:#f8cecc (State)
    style H fill:#d5e8d4 (Renderer)
```

### Layer 1: The DSL Expansion (`keyboard.ts`)
The `d.type()` function is a **Macro**. It doesn't produce one event. It produces dozens.
For the string "A!":
1.  `KEY_DOWN '⇧'` (Shift)
2.  `KEY_DOWN 'A'` -> `TYPE_CHAR 'A'` -> `KEY_UP 'A'`
3.  `KEY_UP '⇧'`
4.  `KEY_DOWN '123'` (Switch to Numbers)
5.  `KEY_DOWN '!'` -> `TYPE_CHAR '!'` -> `KEY_UP '!'`

**The Logic:** `dsl/src/events/keyboard.ts` contains a smart parser that looks ahead. If the next character is `!`, it *knows* it needs the `NUMBERS` layout. It inserts the layout switch events automatically.

### Layer 2: The Core State (`KeyboardState`)
The `world.device.keyboard` object is the single source of truth.
```typescript
interface KeyboardState {
    visible: boolean;
    layout: "qwerty" | "numbers" | "symbols";
    currentKey: string | null; // The key currently being "touched"
    capsLock: boolean;
}
```
When `KEY_DOWN 'A'` fires, `currentKey` becomes `"A"`.
When `KEY_UP` fires, `currentKey` becomes `null`.
This simple state drives the entire visual animation.

### Layer 3: The Renderer (`IOSKeyboard.tsx`)
The renderer is dumb. It just draws the state.
*   **Pop-up Animation:** If `currentKey === "A"`, it renders the iOS-style "Key Pop-up" bubble above the "A" key.
*   **Slide Animation:** We use a `slideProgress` interpolation to animate the keyboard tray sliding up from the bottom (`translateY`).
*   **Predictive Text:** We simulate a static predictive bar (currently "The", "I", "a") to sell the realism.

### Layer 4: The Surface Scaling
The keyboard is designed at **393px** (iPhone 15 Pro width).
However, devices vary in size.
`KeyboardSurface.tsx` wraps the keyboard in a scaling container:
```typescript
scale = targetWidth / 393;
transform: scale(${scale});
transformOrigin: bottom left;
```
This ensures the keyboard looks pixel-perfect on an iPad, a wrapper, or a custom Android frame without rewriting CSS.

## 3. The Typing Engine: Ghost Touches

The realism comes from the "inter-key variance".

### 3.1 WPM & Delays
We define speed presets:
*   **"Fast"**: 2 frames per keystroke (Pro user).
*   **"Normal"**: 3-4 frames (Casual).
*   **"Slow"**: 5+ frames (Grandma).

### 3.2 Seeded Variance
We don't just use `Math.random()`. That would cause the video to flicker between renders.
We use `SeededRNG(seed + textHash)`.
This means "Hello" will *always* have the exact same timing jitter every time you render the video. Determinism is critical for generated video.

### 3.3 Audio Sync
Remember **Lock 1** from Audio Architecture? The keyboard *does not play sounds*.
Instead, the `KEY_DOWN` event (generated by the DSL) is picked up by the **Auto-Sound Engine**.
*   **Rule:** `match: { kind: "KeyboardInput", type: "keyDown" }`
*   **Action:** `PLAY_ONE_SHOT` "keyboard_click_v2.mp3".
This keeps the visuals (key pop-up) and audio (click) strictly synchronized to the same frame.

## 4. Interaction with Apps

How does typing "A" make "A" appear in WhatsApp?
It is a **Two-Step Event Flow**:

1.  **Step 1: The Visual**:
    *   `kind: "KeyboardInput", type: "keyDown", key: "A"`
    *   Effect: Keyboard highlights "A". Sound plays.

2.  **Step 2: The Logical**:
    *   `kind: "KeyboardType", text: "A"`
    *   Effect: **WhatsApp Reducer** listens for `KeyboardType`. It appends "A" to `draft.input.text`.

This separation allows us to have "Ghost Typing" (keys moving but no text appearing, e.g., lagging app) or "Magic Typing" (text appearing without keys moving) if desired, though usually they are paired.

## 5. DSL Authoring Guide

### Scenario A: Standard Typing
"Just type the message."
```typescript
// Includes typing simulation AND text insertion
d.type("Hello world!");
```

### Scenario B: The "Backspace" Correction
Simulating a typo.
```typescript
d.type("Hellp");
d.keyboard.backspace(); // Deletes 'p'
d.type("o world");
```
**Note:** `d.type` handles the capitalization and spacing, but for explicit corrections, use `backspace()`.

### Scenario C: Symbol Switching
"I need to type a complex password."
```typescript
d.type("Pass123!");
```
**Under the hood:** The engine generates:
`Shift` -> `P` -> `a` -> `s` -> `s` -> `123_Key` -> `1` -> `2` -> `3` -> `#+=_Key` -> `!` -> `ABC_Key`.
You don't need to manually switch layouts. The `d.type` macro handles it.

### Scenario D: Manual Key Highlights (Shortcuts)
You want to show "Cmd+C" logic (if we had a desktop keyboard) or just highlight a key for emphasis.
```typescript
d.keyboard.down("space");
d.wait(10);
d.keyboard.up();
```

## 6. Supported Layouts

### 6.1 iOS (QWERTY & Numbers)
We fully simulate the standard iOS English layout.
*   **State:** `visible`, `isPressed`, `layout` (qwerty/numbers).
*   **Special Keys:** `return` (blue), `space` (large), `shift` (toggle).

### 6.2 Android (Material)
*   **Style:** Flat design, different font (Roboto), no key pop-ups (ripple effect instead).
*   **Colors:** Darker grays, distinct "Enter" key style.

To switch keyboards, change the `device` model. The `KeyboardSurface` auto-detects `platform: "ios"` or `"android"`.

## 7. Troubleshooting

### "Keys aren't popping up"
*   **Cause:** Are you using `d.keyboard.setText("Hello")`?
*   **Reason:** `setText` is a debug/magic tool. It instantly sets the value. It avoids the `KEY_DOWN` simulation.
*   **Fix:** Use `d.type("Hello")` for the simulation.

### "Typos in my generated video?"
*   **Cause:** The DSL writes exactly what you tell it.
*   **Feature:** Use this! `d.type("Hlelo")`, `d.backspace()`, `d.backspace()`, `d.type("ello")`. Making AI make mistakes makes it feel more human.

### "Keyboard covers the chat"
*   **Cause:** The layout isn't reacting to keyboard height.
*   **Fix:** Apps typically subscribe to `world.device.keyboard.height`. When `visible` is true, the app should reduce its view height or add bottom padding. WhatsApp and Twitter adapters handle this automatically.
